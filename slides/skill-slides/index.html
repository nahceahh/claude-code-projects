What is a skill?

A skill is a reusable workflow

Think “macro”

Encodes the steps you always repeat: what to inspect, what to output, what to avoid.

Think “policy”

Hard requirements (e.g., don’t run risky commands, don’t create docs) baked into the flow.

Think “interface”

A short name + a trigger = predictable behavior across sessions and projects.

Outcome: higher quality outputs with less prompting.
How it’s invoked

Two ways to trigger

Manual

/<skill-name>

Explicitly run a skill when you want a known playbook.

Automatic

TRIGGER when: ...
DO NOT TRIGGER when: ...

Claude suggests / uses a skill based on context (imports, intent, files).

You ask
“Create a PR” or “use Anthropic SDK”
Skill expands
A longer, structured instruction set is injected
Tools execute
Bash/Read/Edit/etc — in a safe, repeatable order
Where skills live

Local, project, or shared

Shared skills directory

Great for personal playbooks you use everywhere.

Project conventions

Great for “how we do things here”.

Composable

Multiple skills can exist side-by-side (e.g., API, UI, testing, docs).

Why it matters

Skills optimize for reliability

Consistency

Same high-quality process every time (no missed steps).

Safety

Built-in guardrails around risky actions (push, delete, reset).

Speed

Less re-explaining. More doing. Better defaults.

Knowledge capture

Turns tribal knowledge into an executable checklist.

Anatomy

A skill is mostly instructions

TRIGGER when: user asks to create a pull request
DO NOT TRIGGER when: user is just asking questions

Steps:
  1) Inspect status/diff/log
  2) Draft title + summary
  3) Push branch if needed
  4) Create PR with gh

Rules:
  - don't force-push
  - don't bypass hooks
  - ask before destructive ops
Tip: Write skills like you’re coaching a teammate. Make the “order of operations” explicit.
Best practices

Designing good skills

  • Scope tightly: one job, one flow.
  • Tool-first: Read/Grep/Edit before guessing.
  • Safety gates: confirm hard-to-reverse actions.
  • Outputs: specify format (bullets, checklists, code blocks).
  • Triggers: keep them specific to avoid spam.
Rule of thumb: if you had to prompt it twice, it probably deserves a skill.
